## 2. Темы обсуждения

### Часть 1. Интегральные критерии качества SMP Mobile

Подкин показал черновик метрик ([[mobile-quality-metrics]]), прошлись по разделам:

| Группа | Метрики | Комментарий |
|--------|---------|-------------|
| Технические | Cold start (time-to-interactive), кол-во дефектов | В мобилке ценятся; дефекты могут уходить и в поддерживаемость |
| Пользовательские | MAU, среднее число **целевых действий** на пользователя, рейтинг в сторах | Целевые действия = добавление объекта/файла/комментария, редактирование, смена статуса, привязка — не просто «открыл/закрыл форму» |
| Полнота функционала | % покрытия сценариев веб-версии через mobile API | Спорный — см. ниже |
| Поддерживаемость | % покрытия кода автотестами, коэффициент техдолга | iOS-тесты отстают, доработка запланирована на лето |

**Дискуссия по «полноте функционала»:**
- Подкин скептичен: SMP Mobile развивается не активно (усиливаем нативные сильные стороны — камера, пуш, гео, гло; планировали только улучшить оффлайн), показатель будет статичен и мало что скажет о состоянии продукта.
- **Позиция Демьянова:** делать это **интегральным показателем движения**, а не статичным срезом. Складывается по периодам: за период реализовано N фич, добавилось M новых, часть не реализована → переносится. Отражает «жизнь продукта» и утилизацию мобилки продуктами (хватает / не хватает функциональности для закрытия рыночного предложения). Оценивают полноту — продакты, использующие мобилку. Подкин согласился переформулировать.
- По поддерживаемости Демьянов запросил: какие **выводы** даёт покрытие тестами на практике (ловятся ли дефекты до клиента, корреляция «не покрыто автотестами → всплыло у клиента»).

**Договорённость:** Подкин собирает **1-ю итерацию показателей** (с цифрами по текущему состоянию).

### Часть 2. Мультиканальность — предложение подключиться

Демьянов разложил для Подкина предметную область (приглашение поучаствовать со своей командой):

- **Мультиканальность ≠ омниканальность.** Омни — сквозной чат с полным контекстом в любой системе; сейчас этого нет ни в КЦ, ни в Service Desk. Говорим о **мультиканальности** — обработка обращений из разных каналов.
- **Два разных заказчика, две разные мультиканальности:**
  1. **Контактный центр** — высоконагруженный сервис, стандартные коннекторы к каналам (MAX, WhatsApp, Telegram, ВК, Одноклассники, Avito), критичен **виджет на сайте и в мобильном приложении**, обязательная/опциональная аутентификация (формат общения зависит от того, как авторизован пользователь). Приземление в Service Desk. Это «отдельная кухня» (сейчас на legacy через RUDIT).
  2. **Внутренние сервисы / поддержка** — не так нагружена, много самообслуживания (бот, кнопки, мини-апп в мессенджере, регистрация заявок). Каналы: ВК, Telegram, MAX, корпоративные мессенджеры (Mattermost, eXpress).
- **Идея реализации для внутренних сервисов:** не плодить кнопки в чате, а точка входа = чат + «хитрая стройка» с аутентификацией через короткоживущий URL-ключ для текущего пользователя (мобилка/веб-интерфейс прямо в мессенджере). Не обязательно на технологиях NSMP — можно сервер очередей + микросервис + коннектор к NSMP.
- **Предложение Демьянова:** Подкину подключиться, проработать **гипотезу по мультиканальности для внутренних сервисов** (у него в команде есть разработчики/аналитики с опытом по Service Desk / КЦ). Сначала **сфокусироваться на одном типе** мультиканальности. Возможно совместно с виртуальной командой и аналитиками экосистемы.
- **Подкин:** нужно время переварить. Уточнил, что **продуктовых модулей мультиканальности сейчас нет** (только legacy) — то есть это не конкурент текущему модулю.
