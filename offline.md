****Кейс 6****

**Клиент:** Инпас  
**Требование:** Добавить возможность менять статус заявки в офлайн-режиме: взять в работу или закрыть заявку  
**Кейс:** Клиенту важно, чтобы в системе фиксировалось фактическое время совершения действия, а не время успешного разбора действия из очереди синхронизации, чтобы заявка не показывалась как просроченная и не портила KPI.  
  
**Комментарий:**  
В истории объекта, при загрузке файлов и добавлении комментариев сейчас показывается момент синхронизации - показывать только фактическое время совершения действия некорректно.  
Пример: Комментарий, созданный в 12:00 появился в системе не в момент синхронизации (15:00), а в 12:00 и потерялся в списке комментариев - его просто никто не увидел.   
Как вариант, можно создать атрибут или просить клиентов создавать атрибут, в который будет фиксироваться фактическое время выполнения действия.

  

**Варианты решения:** 

1. Форма смены статуса

2. Кнопка ПДПС со скриптом utils.edit для смены статуса с отправкой геопозиции и контекстной переменной queuedData

**Кейсы** 

  

**Недостающие фичи:**

1. Создание объекта (с серверозависимыми атрибутами)
2. Редактирование объекта (с серверозависимыми атрибутами)
3. ПДПС (с серверозависимыми атрибутами)
4. Смена статуса (с серверозависимыми атрибутами)
5. Прикрепление файлов на формах
6. Доступ к списку связанных и вложенных объектов на карточке и их действиям
7. Доступ к базе знаний или инструкциям, доступ к оргструктуре, контактам сотрудника, поиск по списку КЕ
8. Кэширование объектов без их посещения
9. Определение координат сотрудника
10. Фиксация фактического времени смены статуса
11. Уведомление об ошибке синхронизации
12. Отправка сотруднику данных по новым заявкам

**Сложности:**

1. В офлайне не работает рекалк на формах (при изменении типа, Соглашении/услуг, при настроенных скриптах фильтрации и вычисления).
2. Нужно API для подгрузки возможных значений атрибутов СБО, НБО, Агрегирующий, ЭС, НЭС.
3. Мы заранее не знаем, количество значений в списке и не узнаем, пока не загрузим все. В атрибуте может быть много значений, например: 4000.
4. В серверозависимых атрибутах есть пагинация, показывается только 20 объектов, остальные нужно прогружать.
5. Поиск в серверозависимых атрибутах работает только с помощью API. Если объектов будет много, пользователю будет сложно найти нужные. Нужно будет реализовывать поиск через МК
6. Нужно кэшировать переходы во все статусы и формы для каждого перехода
7. Мы можем закэшировать переходы только для текущего статуса. *Если пользователь захочет в офлайне перевести заявку сначала из плана в работу (сможет), а потом из работы в закрыто (не сможет). 
8. На Android сейчас нельзя на форму прикреплять файлы.
9. Если загружать карточки не только из списков объектов, но и из списков связанных и вложенных объектов, рискуем уйти в бесконечную загрузку.
10. Базу знаний не реализовать без поиска и загрузки карточек внутри списков на карточках.
11. Поиск в списке объектов работает только с помощью API и запоминает только ранее выполненные запросы с первыми 20 результатами.



**Груминг, созвон с Ваней, обсуждение с Серёжей**

  

1. Я неправильно поняла решение Вани, думала, что он хочет вынести настройку в офлайна прямо в навигационное меню.

  

2. Решение Вани 

Оставить текущую реализацию как "базовый офлайн-режим" и добавить "расширенный офлайн-режим", т.е. со следующими доработками.

  

В Настройках приложения разместить Настройки офлайн-режима, которые будут включать в себя:

- Список кэшируемых объектов из навигационного меню

- Количество кэшируемых объектов в списках

- Глубину вложенности (карточка или список = 1 уровень вложенности)

- Регулярность актуализации данных   
  

Также в Настройках офлайн-режима будет написан

- размер кэша

- когда последний раз актуализировали данные

  

Вложенность разделов меню нас не интересует.

  

Кэшировать карточку не как сейчас, а полностью - вместе с экранами "Комментарии" и "Файлы".   
У нас уже кэшируются все просмотренные комментарии, но мы почему-то показываем только первые 20.  
У нас уже кэшируются просмотренные файлы, но нет связки, что пользователь смотрел эти файлы на определенной карточке, эту связку нужно добавить.

  

3. Ваня предложил актуализировать данные ночью.

  

4. Я думала о том, чтобы в перспективе отказаться от dbaccess-параметра. Ваня согласен.

  

5. Завела исследование: [NSDWRK-34514 SMP Android. Расход ресурсов устройства при расширенном кэшировании для офлайн-режима](https://naupp.naumen.ru/sd/operator/#uuid:smrmTask$360731627)

  

6. Серёжа за то, чтобы вкладываться и делать добротное решение сразу - настройки офлайна технологом.  
Говорит, это решение будет выглядеть как более качественное, и его легче будет продать клиентам + можно будет заработать деньги за донастройку офлайна специалистами 1-ой линии.  

  

7. Решение с автоматическим кэшированием при входе реализовать можно, но если мы будем кэшировать значения при каждом входе ресурсы устройства будут сильно тратиться.

  

8. Почему нам вообще важны ресурсы устройства: память, батарея?   
Если приложение тратит много ресурсов, сторы могут устроить проверки.

  

9. Ещё есть проблемы/вопросы:   
- Если мы будем кэшировать всё или почти всё, нужно понять что приоритетнее, т.к. есть риск, что часть данных не войдёт в выделенную память.

- Какой объём стоит выделить на кэш расширенного офлайна?
