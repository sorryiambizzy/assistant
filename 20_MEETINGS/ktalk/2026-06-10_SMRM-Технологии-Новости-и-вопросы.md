## Краткое содержание

Еженедельное собрание техофиса SMRM. Три крупных блока: маршрутизация вопросов по документации экосистемы, переупаковка темы обновлений в стратегический трек, и подробный разбор эксперимента Демьянова и Чернякина — встраиваемое приложение бронирования переговорных, сделанное виртуальной командой разработки почти без участия живых разработчиков. Подкин — слушатель, задавал уточняющие вопросы по фидбэку клиента и функциональным требованиям.

## Основные темы

### Документация экосистемы (вопрос Т. Санниковой)
- Татьяна ищет общее видение, куда движется управление знаниями; отдельные активности не складываются в картину.
- **Маршрут:** по любым вопросам экосистемы — сначала к **Севе Шадрину** (ПМ проектов экосистемы), можно заводить заявку в Service Desk.
- **Решение Демьянова:** собрать встречу. Консолидированный заказчик по документации (от продуктов, маркетинга, продаж, сервисного центра) — **Антон Павлов**; владелец экосистемы и зоны документации (внутренней/внешней) — **Всеволод (Сева) Шадрин**. Организацию встречи поставить Шадрину.
- **Минимальный формат первой встречи:** Шадрин, Павлов, Габдрашитова (операционный менеджер направления физ. развития), Санникова.

### Упрощение процесса обновления (В. Кылосов)
- Старт работ состоялся; этап — «понять текущую картину». Вика назначает интервью с участниками процессов обновления; список расширен по подсказке Демьянова (добавить участников по теме комплектов).
- Итог итерации — схема текущего процесса обновления + классифицированный список полей → далее рабочая группа.
- **Решения:**
  - Переименовать задачу в **«Упрощение процесса обновления»**.
  - Зафиксировать как **непрерывное стратегическое направление** развития платформы NSMP («карта намерений»), а не разовый проект — итеративный цикл.
  - Цель (уже сформулирована): «реализовать предсказуемый управляемый процесс выпуска релизов для повышения удовлетворённости и скорости поставки новых версий NSMP». Демьянов просит **добавить метрики-критерии**: обновления быстрые / дешёвые / низкорисковые, с замером прогресса от релиза к релизу (кол-во проблем при обновлении N клиентов).

### Эксперимент: встраиваемое приложение бронирования (Демьянов + Чернякин)
Главная тема встречи.
- **Что сделали:** встраиваемое приложение бронирования переговорных (Calendar Booking) для клиента **Центр 365** (проблемные клиенты в облаке 365), собранное **виртуальной командой разработки** почти без участия живых разработчиков.
- **Подход Демьянова:** взял исходник Чернякина + запись встречи с командой клиента (аналитик СЦ, Лена Свенкова — рук. клиентского сервиса, Андрей Фетисов — рук. Центра 365), прогнал через «команду ассистентов» как кастдев-интервью → дорожная карта, базовая архитектура, постановки, ADR, реализация. Виртуальное агентство безопасности нашло уязвимость (доступ к списку всех бронирований в обход прав) и предложило фикс.
- **Расхождения в подходах с Чернякиным:** настройки (Демьянов — в константах Groovy; Чернякин — конфиг-файл JSON для маппинга, без Key-Value Store); дизайн (Демьянов рисовал в Claude-дизайне по личной подписке → переносил в реализацию, т.к. в Un Design части компонентов нет → кастомные компоненты на React).
- **Фидбэк клиента (на вопрос Подкина):** «очень понравилось, давайте скорее на прод».
- **Артефакты:** исходный код в репозитории SMRM (доступен в Gramax — поиск `Calendar Booking`); сборка стройки (пока **без подписи**, модули не зашифрованы); проектная документация в Gramax (ADR, исследования, требования, аудит безопасности, аудит npm, проверка на 152-ФЗ).
- **Найдена ошибка в документации API:** заполнение атрибута типа «Дата/время» через API регистрации/изменения атрибутов описано неверно (в репозитории примеров нет). Цепочка: Лёша Мурат (последний коммит) → Даня Панишев (подтвердил, что документация неверна). Мурат обещал поправить, но **уходит из компании** — риск, что не успеет.
- **UI 2.0 + low-code (вопрос А. Марковой):** идея вайб-кодинга в UI 2.0 рассматривается, начинать эксперименты можно (Денис Кирпиченков готов). Целевое видение: NSMP даёт надёжный контракт и API, клиент сам собирает фронт ассистентом. **Терминология:** называть не «вайб-кодинг», а **промпт-инжиниринг** — система промптов даёт детерминированный, прогнозируемый результат.
- **Перспектива (Демьянов):** на базе прототипа собрать **шаблон виртуальной команды + инфраструктуру для работы со стройками** (разработка с нуля, помощь клиентам в настройке/развитии). Связанная идея: JSON-RPC / GraphQL-подобный транспорт → для части кейсов (бронирование) бэк может быть не нужен, всё на универсальном контракте + фронт. Чернякин: тему «без бэкенда» нужно обсудить отдельно. Интерес проявил Project Ruler (Аня Лаптева) — у них тоже задача бронирования.

## Решения
- По документации экосистемы — собрать встречу (Шадрин организует): Шадрин, Павлов, Габдрашитова, Санникова.
- Тему обновлений переименовать в «Упрощение процесса обновления» и вести как непрерывное стратегическое направление с метриками.
- Прототип Calendar Booking → на ревью (Чернякин как владелец процесса + Кирилл Егоров как главный разработчик), затем бизнес-аналитика и сбор требований перед продакшеном.
