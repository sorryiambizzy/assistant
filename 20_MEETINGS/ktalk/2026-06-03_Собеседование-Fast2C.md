# Собеседование — экс-проджект Fast2C

**Дата:** 03.06.2026 14:05
**Длительность:** ~50 минут
**Платформа:** Kontur Talk
**ID записи:** `vTghZ9c50M4B7Jt9RRgL`

> Кандидат — Илья, руководитель проекта (проджект) продукта **Fast2C** (он же FastTisi/FastOTC из протокола 26.05) — CRM-сервиса для криптоброкеров. Ушёл вместе со всей командой в феврале 2026. Собес совмещал разведку рынка/конкурента и оценку кандидата.

---

## Ключевое содержание

### Команда и стек
- Кандидат — проджект, ушёл с командой в феврале 2026.
- Доступны 2 разработчика, шарят в скрипте, работали на проекте ~1,5 года: **Богдан** (ведущий разраб, сейчас в Грузии, не работает) и **Степан** (Питер, статус не уточнял). Резюме обещал прислать.
- Стек: фронт — JS, бэк — PHP, фуллстек. Сервис написан поверх движка, который разрабатывала другая команда владельца (Дмитрия) для своих нужд.

### Продукт и клиенты
- Fast2C — сервис для **брокеров**, не обменников. Брокер: долгосрочные отношения с клиентом, крупные переводы, работа в разных странах (принёс рубли в Москве — получил USDT/доллары на Пхукете). Обменник: много мелких операций для физиков, всё автоматом (фикс-курс с коэффициентами).
- На пике — **4 платящих клиента** (компании-брокеры), ~5-6 в тестовом периоде.
- Рынок: ~**200 компаний-брокеров**, сидят в Москва-Сити, все друг друга знают → привлечение только сарафан, без рекламы. (Совпадает с оценкой Наумен.)

### Ценообразование
- Месячная подписка, тариф **~$800–3000/мес**.
- Брали деньги **за транзакции**, где транзакция = запись в БД (одна сделка = несколько транзакций). Для клиента непрозрачно, тяжело рассчитать тариф — слабое место.

### Ключевые ошибки / уроки
- Классическая стартаперская ошибка: >1 года пилили продукт под **1 продвинутого брокера-партнёра**, не проведя анализ рынка и категоризацию клиентов.
- Наворотили избыточный функционал (автомэтчинг сделок с расчётом маржинальности/профита), который рынку оказался не нужен. Последние ~4 месяца только **резали** фичи (появились версии pro / lite / ещё lighter).
- Проблемы масштабирования: монолит, архитектура «на коленке», пересчёт не периодами, а всего сразу → при росте объёма данных тормозило. Для брокеров скорость критична.
- Один клиент на экзит-интервью ушёл обратно в **свой Telegram-бот** («быстро, понятно, больше не надо»). Авторасчёт маржи/дашборд не стал достаточной ценностью, чтобы уйти из телеги.
- Делали Telegram-бот + miniapp; miniapp тоже тормозил → оставили просто бота. По боту много запросов на доработку = им пользуются.

### Почему проект «умер»
- Владелец **Дмитрий** (IT-бизнесмен, сильный программист, авторитарный максималист) потерял интерес; команда съела бюджет; клиентов мало; непонятно, что делать дальше, кроме v2.0 (идея «мини-рынка» — брокеры Fast2C мэтчат сделки между собой).
- Сейчас зомби-режим: обслуживают существующих платящих, разработки нет. Дмитрий хотел уходить в DeFi.

### Роль кандидата (проджект)
- Функцию продакта по факту не вёл (признаёт сам — размазана между ним и Дмитрием). Учился на продакта, но «застрял одной ногой здесь, другой там».
- Сильная сторона — выстраивание процесса и команды: из «гаражного стартапа» с релизами по воскресеньям (Дмитрий) вывел на предсказуемые еженедельные релизы — внедрил QA, регресс, чеклисты, релизную модель, ветки, кроссфункциональность. Команда до 7 человек. Наймом не занимался (только увольнениями — Дмитрий нанимал авторитарно).

### Вердикт Михаила
- На продакта — **не подходит** (нет системной продуктовой функции; информация с рынка шла не ему, а внешним людям).
- Возможно — РП на небольшие/средние проекты; включили в базу Наумен.
- Многое из рассказа кандидата **подтверждает рыночный скепсис Наумен** по криптоCRM.
- Вакансия «полу-вакансия» — Наумен ещё решает, открывать ли её.
- Отметили открытость и честность кандидата.

### Договорённости
- Кандидат пришлёт: резюме 2 разработчиков, статистику по работе брокеров (число сделок), список конкурентов, детали ценообразования (уточнит цифры).
- Кандидат спросит у Дмитрия разрешения дать его контакт (email) — Наумен хочет пообщаться с Дмитрием.
- Вся коммуникация через Евгения; при подвижках — вывод на HR.
