## Контекст встречи

Клиент использует SMP Mobile для выездных инженеров в мониторинговом бизнесе (ITAM). Инженеры работают в удалённых местностях с нестабильной связью (Саха, Якутия, Хабаровск, Дагестан — 300 км газопровод). У клиента 15 из 25 лицензий — процессорные (вахтовый метод работы).

Конкурент **ОГДСК** активно наращивает позиции, добавил оффлайн-режим. Кирилл Вац (предположительно из Нау) ушёл в ОГДСК.

## Проблемы клиента

### 1. Отсутствие группировки и фильтрации

**Суть проблемы:**
- Нет возможности группировать активы и задачи по атрибутам
- Инженеры не могут быстро оценить количество объектов определённого типа
- Сложно находить задачи по ответственному, объекту, виду работ

**Текущий сценарий:**
- Инженеры выгружают в Excel и вручную группируют данные
- Тратят время на листание длинных списков

**Желаемое решение:**
- Вложенный список с возможностью конфигурации на уровне интегратора
- Примеры группировок:
  - Активы: Тип → Марка → Модель
  - Задачи: Область → Месторождение → Объект
  - Задачи: Вид работ → Область → Задачи
  - Задачи: Ответственный → Объект
- Отображение количества вложенных элементов
- Возможность создавать несколько предустановок группировки

### 2. Проблемы с оффлайн-режимом

**Суть проблемы:**
- При отсутствии связи мобильное приложение не загружается
- Ограничение на кэш (~20 задач)
- Невозможность менять статус задач в оффлайне
- Задержки с отправкой документов до 10 дней

**Текущий сценарий:**
- Инженер последний раз отчитался 6 апреля (встреча 16 апреля)
- Задержки по документам → кредитование заказчиков
- Бизнес-процессы блокируются

**Желаемое решение:**
1. Кэширование всех задач с важными статусами (конфигурируется)
2. Возможность менять статус в оффлайне с визуальной индикацией очереди синхронизации
3. Подсветка задач, находящихся в очереди на синхронизацию
4. Кнопка "Загрузить всё" для принудительной загрузки
5. Размер кэша до 900 МБ (клиент готов покупать отдельные телефоны)

### 3. Отсутствие чеклистов с валидацией

**Суть проблемы:**
- Нет обязательных чеклистов с проверкой выполнения
- Нет обязательных полей (например, пробег при ТО)
- Нет контроля количества фотографий

**Текущий сценарий:**
- Клиент реализовал функционал самостоятельно через 1С и Битрикс
- Боится, что при обновлении Naumen всё сломается

**Желаемое решение:**
- Обязательные чеклисты с контролем выполнения
- Обязательные поля при определённых действиях
- Контроль количества фотографий (например, 3 ракурса оборудования)

### 4. Юзабилити проблемы

**Фотографии:**
- Сейчас: одно фото → возвращение → повторное действие
- Желаемо: сделать несколько фото → выбрать → удалить ненужное → отправить

**Обновление списка задач:**
- Сейчас: при нажатии "Назад" список обновляется и накидывает наверх
- Желаемо: сохранить позицию, обновление по кнопке или свайпу вниз

**Ручная подпись:**
- Желаемо: заказчик расписывается на телефоне, делается 2-3 фото
- Встроенное приложение "Ручная подпись" найти не удалось

## Сценарии использования

### Работа выездного инженера

1. **Получение задач:**
   - Инженер видит 30-40 задач на вахту
   - Группировка по области/объекту/виду работ

2. **На объекте:**
   - Открыть задачу
   - Посмотреть предыдущие наряды по ТС
   - Заполнить чеклист (диагностика или фотоотчёт)
   - Выбрать оказанные услуги из договора
   - Указать количество (например, 2 датчика)
   - Прикрепить фотографии
   - Завершить и отправить на проверку

3. **Сверка активов:**
   - Инженеры сверяют оборудование между сменами
   - Перенос актива от одного инженера к другому
   - Обязательное подтверждение принятия

### Контекст бизнеса

- Холдинги типа "Газстройпром" — задачи от разных юрлиц на один объект
- Вахтовый метод работы — массовая смена ответственного при смене вахты
- Смена ответственного на объекте — поиск всех задач этого объекта
- Выписка актов (PDF) из задач
