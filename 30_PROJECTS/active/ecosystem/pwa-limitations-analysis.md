## 🔍 Детальный анализ ограничений PWA

### 1. Почему Service Worker не заменяет нативную БД?

| Аспект | Service Worker + Cache API | Нативная локальная БД |
|--------|---------------------------|----------------------|
| **Хранилище** | Cache API (HTTP-ответы), IndexedDB (до 50-60% от диска) | SQLite/CoreData (неограниченно в пределах приложения) |
| **Структура данных** | Cache API — по URL (ключ-значение). IndexedDB — NoSQL (документы, индексы) | Полноценные реляционные связи, индексы, транзакции |
| **Архитектура бэкенда** | Не зависит от клиентского хранилища (может быть SQL/NoSQL) | Не зависит от клиентского хранилища |
| **Синхронизация** | Background Sync API (экспериментально на iOS) | Надежная фоновая синхронизация |
| **Сложность запросов (клиент)** | IndexedDB - NoSQL на клиенте, ограниченные офлайн-запросы | SQL - JOIN, агрегации, сложные выборки на клиенте |
| **Миграции** | Ручная обработка версий схемы | Встроенные механизмы миграций |
| **Производительность** | Ограничена JavaScript runtime | Нативная производительность |

**Влияние на Job №1 (Офлайн-работа):**

```
Сценарий: Выездной специалист в зоне без интернета

Нативное приложение:
├── Полноценная локальная БД со всеми данными
├── Сложные выборки и фильтрация офлайн
├── Надежная очередь изменений для синхронизации
└── Пользователь работает как обычно

PWA:
├── Ограниченный кеш ресурсов (UI, шаблоны)
├── IndexedDB для данных (NoSQL-подобное хранилище на клиенте)
├── Архитектура бэкенда не меняется (может быть SQL)
├── Риск потери данных при переполнении кеша
└── Синхронизация только при открытии приложения
```

> **Важное уточнение:** IndexedDB — это формат хранения на *клиенте* (в браузере). Архитектура бэкенда PWA может быть любой — PostgreSQL, MongoDB, микросервисы. Это не влияет на выбор локального хранилища. IndexedDB используется только для офлайн-кеша на устройстве пользователя.

**Практические ограничения:**
- На iOS Safari_quota (~50-500MB на домен)
- Сложность реализации конфликтов синхронизации
- Нет гарантии выполнения Background Sync (iOS не поддерживает)
- При очистке кеша браузера - данные теряются

📖 **Примеры реализации:** [[30_PROJECTS/active/ecosystem/pwa-implementation-guide.md#кеширование-списков-и-карточек|PWA Implementation Guide: Кеширование]]

### 2. Фоновая геолокация: как ограничена и влияние на сценарии

| Возможность | Нативное приложение | PWA |
|-------------|-------------------|-----|
| **Геолокация при активном приложении** | ✅ | ✅ |
| **Геолокация в фоне (свернуто)** | ✅ | ❌ |
| **Геолокация при закрытом приложении** | ✅ (significant location change) | ❌ |
| **Точность** | Высокая (GPS + Wi-Fi + cellular) | Средняя (только при активном приложении) |
| **Потребление энергии** | Оптимизировано | Не применимо (нет фона) |

**Влияние на Job №2 (Планирование выездов и маршруты):**

```
Сценарий: Диспетчер видит местоположение специалистов на карте

Нативное приложение:
├── Фоновая передача координат каждые N минут
├── Отображение реального времени на карте
├── Уведомление при прибытии на точку (геофencing)
└── Диспетчер видит актуальное положение

PWA:
├── Координаты передаются только пока приложение открыто
├── При сворачивании - передача прекращается
├── Геофencing невозможен
└── Диспетчер видит устаревшие данные
```

**Технические ограничения:**
- Geolocation API требует активной вкладки
- Нет Wakelock для поддержания работы в фоне
- Background Geolocation API не поддерживается iOS Safari
- Geofencing API отсутствует в браузерах

**Влияние на Jobs:**
- **№2** - Невозможно автоматически подтверждать прибытие на точку
- **№2** - Диспетчер не видит реальное положение специалистов
- **№2** - Построение оптимальных маршрутов затруднено

### 3. Почему невозможна запись видео в PWA?

| Операция | Нативное приложение | PWA |
|----------|-------------------|-----|
| **Съемка фото** | ✅ | ✅ (через `<input type="file" capture>`) |
| **Запись видео** | ✅ (Camera API) | ❌ (MediaRecorder работает ограниченно) |
| **Доступ к галерее** | ✅ | ✅ (через input) |
| **Редактирование видео** | ✅ | ❌ |
| **Сжатие перед отправкой** | ✅ | Ограниченно |
| **Фронтальная/задняя камера** | ✅ Полный контроль | ⚠️ Ограниченный |

**Влияние на Job №3 (Фото/видео фиксация):**

```
Сценарий: Выездной специалист фиксирует выполненную работу

Нативное приложение:
├── Запись видео прямо в приложении
├── Предпросмотр и удаление неудачных кадров
├── Сжатие и отправка больших файлов
└── Полный контроль над качеством и форматом

PWA:
├── Только через системное приложение камеры
├── Запись video происходит в системное приложение
├── После записи - возврат в PWA для выбора файла
├── Пользовательский опыт разрывается
└── Нет контроля над процессом записи
```

**Технические ограничения:**
- `MediaRecorder` API не поддерживается Safari iOS (вплоть до 2026)
- На Android Chrome поддерживается, но без гибкости натива
- Нет доступа к параметрам записи (битрейт, кодек)
- Нет прямого потока с камеры для обработки
- Нет доступа к системному альбому для сохранения

**Работающие альтернативы (с оговорками):**
- `<input type="file" accept="video/*" capture>` - открывает системное приложение камеры
- Пользователь записывает видео в системной камере
- Затем возвращается в PWA для прикрепления файла

**Почему это проблема для Job №3:**
- Разрыв пользовательского опыта (context switch)
- Нет предпросмотра прямо в форме
- Невозможность добавить примечания во время записи
- Нет контроля над качеством и размером файла

### 4. Ограничения Push-уведомлений на iOS

| Параметр | Android PWA | iOS PWA | iOS Натив |
|----------|-------------|---------|-----------|
| **Push-уведомления** | ✅ (FCM) | ⚠️ (ограничены) | ✅ (APNs) |
| **Требование для push** | Service Worker + уведомление | PWA добавлен на Home Screen | Любое приложение |
| **Badge на иконке** | ✅ | ❌ | ✅ |
| **Звук уведомления** | ✅ | Ограничен | ✅ |
| **Action buttons** | ✅ | ❌ | ✅ |
| **Группировка** | ✅ | Ограничена | ✅ |
| **Бесшумные push** | ✅ | ❌ | ✅ |
| **Vibration** | ✅ | ❌ | ✅ |

**Требования для Push на iOS PWA:**

```
Условия работы Push на iOS Safari:
1. PWA должен быть добавлен на Home Screen (не просто открыт в браузере)
2. Пользователь должен явно разрешить уведомления
3. Уведомления показываются только если PWA добавлен на домашний экран
4. Нет badge count на иконке приложения
```

**Влияние на Job №4 (Push-уведомления):**

```
Сценарий: Специалисту приходит срочное уведомление о новой заявке

Android PWA:
├── Push приходит корректно
├── Можно добавить action buttons
├── При клике - открытие PWA
└── Опыт приближен к нативному

iOS PWA:
├── Push работает только если PWA на Home Screen
├── Пользователь должен понимать и выполнить добавление на экран
├── Нет badge count (не видно количество уведомлений)
├── Нет action buttons (нельзя "Принять"/"Отклонить" из уведомления)
├── Нет вибрации и кастомных звуков
└── Многие пользователи не добавляют PWA на Home Screen

iOS Нативное:
├── Push приходит всегда
├── Badge count показывает количество
├── Action buttons для быстрого действия
├── Кастомные звуки и вибрация
└── Максимальная информативность
```

**Технические ограничения:**
- Web Push API на iOS Safari требует PWA на Home Screen
- Уведомления не приходят если просто открыть PWA в браузере
- Отсутствие badge count - пользователь не видит количество пропущенных
- Нет возможности rich notifications (images, buttons, inputs)
- Нет silent push для фоновой синхронизации данных

**Практическое влияние на пользователей:**
- Пользователи привыкли к badge count и могут пропустить уведомления
- Action buttons критичны для быстрого реагирования (Job №4)
- Отсутствие вибрации/звука снижает заметность
- Для работы push требуется дополнительное действие пользователя (add to Home Screen)

> **Сценарий:** Выездной специалист скачал PWA и поехал в зону с плохим покрытием интернета

### Краткий ответ на 4 ключевых вопроса:

| # | Вопрос | Ответ | Технически |
|---|--------|-------|------------|
| 1 | Закэшировать списки объектов (заявки, задачи, оборудование)? | ✅ **Да** | IndexedDB или Cache API |
| 2 | Закэшировать карточки объектов с детализацией? | ✅ **Да** | IndexedDB или Cache API |
| 3 | Закэшировать настройки динамических форм? | ✅ **Да** | IndexedDB (метаданные форм) |
| 4 | Заполнить заявку со справочниками (сотрудники, оборудование)? | ✅ **Да** | IndexedDB (справочники кешируются) |

├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ПРИ ИНТЕРНЕТЕ (подготовка к выезду):                              │

│  ┌─────────────┐    Очередь изменений  ┌─────────────┐          │
│  │ IndexedDB   │ ─────────────────────→ │   Server    │          │
│  │ (pending)   │                        │             │          │
│  └─────────────┘                        └─────────────┘          │

### Что НЕ работает офлайн (важно понимать)

| Операция | Почему |
|----------|--------|
| 📝 Новые справочники (новые сотрудники добавили сегодня) | Не пришли с сервера |
| ✅ Проверка уникальности на сервере | Нет соединения |
| 🔍 Автоподстановка при вводе (если не закэшировано) | Нет доступа к серверу |
| 📤 Загрузка новых файлов на сервер | Можно выбрать, но отправка позже |
| 🔄 Обновление статуса другими пользователями | Не видно в реальном времени |

| Сценарий | PWA может | Нюансы |
|----------|-----------|--------|
| ✅ Чтение списков офлайн | Да | Данные актуальны на момент последней синхронизации |
| ✅ Чтение карточек офлайн | Да | Кешируются по мере открытия или пакетно |
| ✅ Создание объектов офлайн | Да | Сохраняется локально с `pending_sync` |
| ✅ Редактирование офлайн | Да | Изменения в очереди на синхронизацию |
| ✅ Ссылки на справочники | Да | Только закэшированные значения |
| ⚠️ Валидация на сервере | Частично | Офлайн только клиентская валидация |
| ⚠️ Поиск по всем данным | Ограниченно | Только по закэшированному |
| ⚠️ Конфликты | Требует обработки | Нужна стратегия разрешения |

## 📦 Предварительное кеширование (Pre-caching)

> **Сценарий:** Пользователь понимает, что едет в зону без интернета и заблаговременно кэширует все данные

### Что можно кешировать

| Тип данных | Кешируется | Технология | Ограничения |
|------------|------------|------------|-------------|
| **Списки объектов** | ✅ Да | IndexedDB / Cache API | Размер хранилища |
| **Карточки объектов** | ✅ Да | IndexedDB / Cache API | Размер хранилища |
| **Динамические формы** | ✅ Да | IndexedDB | Небольшой размер |
| **Справочники** | ✅ Да | IndexedDB | Обычно небольшой |
| **Прикрепленные файлы** | ⚠️ Да | IndexedDB / Cache API | Размер — критично |
| **Изображения** | ✅ Да | Cache API | Можно сжать |
| **Видео** | ⚠️ Ограниченно | Cache API | Большой размер |
| **Карты (тайлы)** | ✅ Да | Cache API | Ограничить область |

📖 **Примеры реализации:** [[30_PROJECTS/active/ecosystem/pwa-implementation-guide.md#предварительное-кеширование|PWA Implementation Guide]]

### Ограничения и рекомендации

| Ограничение | Значение | Рекомендация |
|-------------|----------|--------------|
| **Макс. размер (iOS Safari)** | ~50-500MB на домен | Сжимать данные, не кешировать видео |
| **Время загрузки** | Зависит от объема | Показывать прогресс, дать возможность отмены |
| **Устаревание данных** | Неизвестно на клиенте | Показывать время последнего обновления |
| **Очистка кеша браузером** | Возможна пользователем | Дублировать критические данные |
| **Смена данных на сервере** | Не видна офлайн-клиенту | Проверять версию при синхронизации |

## ❌ Задачи, НЕ покрываемые PWA

### №13 (🟡 Средний приоритет)
> **Выездной специалист / Менеджер услуг:** Когда нужно перераспределить работу, я хочу быстро назначать ответственного или брать объект в работу самому, чтобы не тормозить процесс.

**Причина:** VoIP звонки через PWA невозможны. Для быстрой координации часто требуется именно голосовая связь.

### №17 (🟢 Низкий приоритет)
> **Инициатор обращений / Специалист поддержки:** Когда мне нужно обсудить объект системы с командой, я хочу читать и писать комментарии с поддержкой упоминаний и ответов, чтобы уточнять контекст обсуждения.

**Причина:** Комбинированное требование VoIP звонков + STT/TTS для голосовых комментариев.

## ⚠️ Задачи с ЧАСТИЧНЫМ покрытием PWA (требуют внимания)

### Критические для выездных специалистов:

- **№1** - Офлайн-работа (нужна надежная синхронизация)
- **№2** - Геолокация и маршруты (фоновая трекировка ограничена)
- **№3** - Фото/видео фиксация (видео-запись невозможна)

### Критические для уведомлений:
- **№4** - Push-уведомления (на iOS ограничения)
- **№5** - Авторизация (WebAuthn менее удобен)

### Связанные с расширенными возможностями:
- **№8** - App Clips/Instant Apps (мгновенный доступ без установки)
- **№9** - Виджеты на главном экране
- **№12** - App Clips/Instant Apps для быстрого доступа
- **№23** - App Clips/Instant Apps для регистрации
