## Ключевые решения

### 1. Ценности SDK 2.0 для клиентов ✅

**Решение:** Сформировать список ценностей для мотивации миграции.

**Ценности:**
- **Дизайн-система:** Клиент предоставляет 23 цвета → SDK полностью перекрашивается
- **Автотесты:** 815 тестов на Android, покрытие повышает стабильность
- **API для кастомизации:** Возможность реализовать кнопки, баблы, input bar по-своему
- **Тесты для клиента:** Крючки API для написания интеграционных тестов перед релизом
- **Документация:** Markdown-формат, AI-агенты для внедрения, Migration Guide

**Обоснование:** Эти аргументы будут использоваться в переговорах с клиентами о миграции.

**Ответственный:** Иван Портнов

---

### 2. Проблемы SDK 1.0 ✅

**Решение:** Документировать проблемы SDK 1.0 для создания контраста с SDK 2.0.

**Проблемы:**
- ❌ Нет автотестов → нестабильность при новых фичах
- ❌ Долго разрабатывать и страшно чинить
- ❌ Устаревший UI (нет свайпов, нет современной внешности)
- ❌ Сложная кастомизация (каждый элемент отдельно)
- ❌ Нет API для полной кастомизации компонентов

**Обоснование:** Клиенты должны понимать, что SDK 1.0 технически устарел.

**Ответственный:** Иван Портнов

### 3. Запрошенные но нереализованные функции ⚠️

**Решение:** Собрать список функций, которые клиенты просили, но не были реализованы.

**Функции:**
- **Сжатие картинок** — ✅ Реализовано для Магнита
- **Контекстные кнопки** — 🔄 В планах (автоматическая генерация по запросу)
- **Кастомизация кнопок** — ✅ В SDK 2.0 через API (Магнит: большие названия, обрезание текста)
- **Аналитика поведения пользователя** — ❌ Не в MVP

**Магнит (документы):**
- Кнопки в разных режимах отображения
- Названия заказов с датами в кнопках
- Обрезание текста в кнопках

**ЦБ:** Требуется уточнение у Дениса

**Ответственные:**

### 4. Стратегия миграции ✅

**Решение:** Предложить A/B тестирование как рекомендуемый путь миграции.

**Процесс:**
1. Демо-приложение в магазинах (Google Play, App Store)
2. Подключение к корпоративному чат-серверу клиента
3. Интеграция SDK 2.0 в тестовую версию МП
4. A/B тестирование (Feature Toggle)
5. Постепенный rollback на SDK 2.0

**Возможности A/B:**
- Встроить обе версии SDK 1 и SDK 2 в одно приложение
- Переключение через настройку на сервере
- Рекомендуемый, но не обязательный путь

**Обоснование:** Снижение рисков при миграции, возможность быстрого отката.

**Ответственный:** Иван Портнов

---

### 5. Демо-приложение ✅

**Решение:** Выпустить демо-приложение в магазины.

**Функционал демо:**
- Подключение к демо-стенду Naumen (боты, живые ответы)
- Возможность подключения к корпоративному серверу клиента
- Демонстрация всего функционала: реплай, фото, завершение диалогов
- Опционально: настройка цветовой темы (3-5 дней работы)

**Обоснование:** Пресейлы смогут показывать SDK на устройстве клиента, а не через TestFlight.

**Ответственный:** Иван Портнов

**Срок:** 3-5 дней для базовой версии, +3-5 дней для настройки тем

---

### 6. Совместимость с чат-сервером ⚠️

**Решение:** Выяснить минимальную версию чат-сервера для SDK 2.0.

**Текущий статус:**
- SDK 2.0 использует Chat Server API v2
- API v2 существует много лет (для виджета на сайт)
- Вероятно, большинство клиентов уже на совместимой версии
- Возможна установка второго инстанса чат-сервера параллельно

**Действия:**
- [ ] Выяснить с какой версии существует API v2
- [ ] Документировать требования к чат-серверу
- [ ] Проверить совместимость у текущих клиентов

**Ответственный:** Иван Портнов

---

### 7. Лицензирование ✅

**Решение:** Ввести две модели лицензирования.

**Модели:**
1. **Новая лицензия** на SDK 2.0 — полная стоимость
2. **License Upgrade** — для владельцев SDK 1.0, дешевле в ~2 раза

**Обоснование:** Психологическая комфортность для существующих клиентов — не платить полную цену.

**Ответственный:** Алексей Гусев
