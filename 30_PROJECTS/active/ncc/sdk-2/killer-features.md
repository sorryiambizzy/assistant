# NCC — Chat SDK 2.0. Киллер-фичи

> Родительский проект: [[30_PROJECTS/active/ncc/sdk-2/sdk-2|NCC — Chat SDK 2.0]]
> Реестр фич/идей для клиентов также частично ведётся в [[30_PROJECTS/active/ncc/sdk-2/sdk-2#7.3 Идеи фич/технологий для Магнита|sdk-2.md §7.3]].

Документ собирает «киллер-фичи» SDK 2.0 — функционал, который даёт прямую ценность интеграторам (Магнит, ОТП и др.) и отличает SDK 2.0 от SDK 1. Для каждой фичи: как было в SDK 1, что хочет клиент, предложение по реализации в SDK 2.0.

---

## 1. Счётчик непрочитанных сообщений (unread badge)

> ⚠️ **Аналитика не готова — требуется сделать.**

**Заказчики, использующие фичу:** ЦБ, Магнит.

**Запрос клиента:** клиент хочет получать уведомление, когда пришло новое сообщение от оператора, чтобы подсветить в табе бейдж с количеством непрочитанных сообщений.

**Как было в SDK 1:**
- Просто метод API — клиент сам дёргал «получить количество непрочитанных».
- Минусы: pull-модель, нет реактивности; чтобы держать бейдж актуальным, клиенту приходилось опрашивать SDK, а это лишний код на стороне интегратора и задержка обновления.

**Предложение для SDK 2.0 — перейти с pull на push:**
- Отдавать счётчик непрочитанных через **подписку/колбэк** (observable / listener / Flow на Android, publisher/делегат на iOS), а не только разовым методом.
- SDK сам пушит обновление, когда:
  - пришло новое сообщение от оператора (счётчик +N);
  - пользователь прочитал сообщения (счётчик пересчитывается/обнуляется).
- Бейдж в табе у интегратора становится «тупым» отображением значения из подписки — клиентский код минимален.
- **Важно:** счётчик должен обновляться даже когда экран чата закрыт (фоновое получение / по событию push-уведомления), иначе бейдж не загорится в момент прихода сообщения.

**Открытые вопросы:**
- Оставлять ли разовый метод-геттер для обратной совместимости и «холодного» старта (получить текущее значение при инициализации)? — скорее да, как дополнение к подписке.
- Источник правды для счётчика при работе с push: считать локально по входящим событиям или сверять с сервером.
- Поведение в многооконных/многотабовых сценариях клиента.

## 2. Стартовое сообщение от имени пользователя при открытии чата

> ⚠️ **Аналитика не готова — требуется сделать.**

**Запрос клиента:** у ОТП точно есть функционал отправки сообщения **от имени пользователя в момент открытия чата**. Используется, когда клиента перекидывают из какого-либо раздела приложения в чат, чтобы автоматически стартануть диалог и вывести пользователя на оператора.

**Сценарий (JTBD):**
- Пользователь находится в разделе приложения (например, «оплата», «заказ», «поддержка по продукту»).
- Нажимает «написать в чат» → приложение открывает чат SDK и **сразу отправляет заранее заданное сообщение от имени пользователя** (например, «Здравствуйте, у меня вопрос по заказу №123»).
- Это сообщение запускает диалог и инициирует маршрутизацию на оператора — пользователю не нужно ничего печатать, чтобы попасть в очередь.

**Предложение для SDK 2.0:**
- Параметр инициализации/открытия чата: **initial/prefilled message**, который SDK отправляет от имени пользователя сразу после установления сессии.
- Поддержать варианты:
  - **видимое** сообщение в ленте (пользователь видит, что он «написал»);
  - опционально **скрытое/служебное** сообщение для маршрутизации (контекст раздела, ID сущности) — чтобы оператор/роутинг получил контекст, но лента не засорялась. Решить, нужен ли скрытый вариант.
- Передавать вместе со стартовым сообщением **контекст-атрибуты** (откуда пришёл пользователь, ID заказа/раздела) — переиспользовать механизм дополнительных атрибутов из push/панели ввода (NCCPRD-693 / NCCPRD-692).
- Гарантировать, что автоотправка происходит **один раз** на открытие, а не повторяется при реконнекте/возврате на экран.

**Открытые вопросы:**
- Где задаётся текст: на стороне приложения (интегратор) или конфигурируется на сервере по точке входа?
- Нужна ли отправка стартового сообщения только при **первом** открытии диалога или каждый раз при заходе из раздела.
- Как это сочетается с маршрутизацией на оператора на стороне Naumen (тематизация/очереди).

## Статус и приоритизация (09.06.2026)

> Источник: [[30_PROJECTS/active/ncc/sdk-2/meetings/2026-06-09-mobile-dev-service|синк с Ликулиным 09.06]]. Состав фич определяет А. Гусев — в отпуске **до 16.06**.

- **«Стартовое/приветственное сообщение» (§2) — единственная готовая к проработке** киллер-фича на данный момент. Постановка **не провалидирована** Иваном, возможны доработки на стороне сервера. Ссылка на тикет для постановки **аналитике** (критический путь) **передана Александру на синке 09.06** ✅.
- **Сроки киллер-фич:** реалистично **июль–август**, но только после определения состава + аналитики. Аналитик один — на приветственное сообщение и на звонилку.
- **Поддержка iOS 26 (Liquid Glass)** обсуждалась как мотиватор перехода на SDK 2, но Ликулин **не считает её киллер-фичей**: «клиенты не любят платить за нефункциональные доработки». Задача поддержки iOS 26 в SDK 1 — **на HOLD** до выработки стратегии (детали — в файле синка §6).
