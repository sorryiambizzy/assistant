# Skill vs Plugin: начать со skill

**Дата:** 04.06.2026
**Статус:** принято

## Контекст

Для индексированного поиска по доке Naumen внутри Claude Code есть два варианта оформления:
- **Skill** — markdown-файл с инструкцией и триггерами, использует существующие инструменты (curl, qmd, Bash).
- **Plugin** — пакет с собственными slash-командами, hook'ами, опционально собственным MCP-сервером.

## Решение

Стартуем со **skill**. Плагин — позже, если/когда подъём оправдан.

## Аргументы

- Skill = 1 файл, готов за день. Плагин = манифест + версионирование + публикация в marketplace, дни-недели.
- На discovery нам важна скорость итераций по содержанию, а не упаковка.
- Skill можно скопировать коллеге одной командой (`cp -r ~/.claude/skills/naumen-docs <их-папка>`); marketplace нужен только при росте аудитории.
- Реальная польза проверяется содержанием инструкции и качеством индекса, а не формой доставки.

## Триггеры пересмотра (когда переходить на плагин)

- Аудитория >5 человек → нужен централизованный канал доставки/обновлений.
- Появилась необходимость в shared индексе/MCP-сервере (не у каждого свой `~/.cache/qmd/`).
- Нужны slash-команды (`/naumen-find`, `/naumen-cite`) или hooks (например, автоматический поиск при упоминании названия страницы).
- Какая-то функция упёрлась в ограничения skill-формата.

## Альтернативы, которые рассмотрели

- **Плагин сразу** — отброшен из-за времени до first value.
- **Скрипт без интеграции в Claude Code** — отброшен, теряется удобство для аналитика, работающего в Claude Code.
