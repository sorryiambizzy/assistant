# Multichannel — разбор экранов демки

Разбор макета мобильного приложения IT-поддержки (3 экрана) по проекту [[30_PROJECTS/active/SMRM/multichannel/multichannel|Multichannel]]. Сценарий-пример: заявка «VPN не подключается с рабочего ноутбука».

---
## Концепция

Концепция проекта — единый модуль мультиканальности (работа из нескольких каналов, переход между ними без потери контекста). Полное описание каналов и ценности для пользователя и компании-клиента — см. [[30_PROJECTS/active/SMRM/multichannel/multichannel#2. Концепция проекта|раздел «Концепция проекта» в карточке проекта]].

Разбираемые ниже экраны описывают сам **SDK** — единое ядро, которое переиспользуется во всех каналах (нативное приложение ITSM как оболочка для SDK, SDK в приложении клиента, Telegram mini apps, MAX, Mattermost). За счёт общего SDK пользовательский опыт во всех каналах одинаковый.

---

## Общие требования к пользовательскому опыту

Требования распространяются на все экраны SDK и на все каналы, в которых он встраивается.

- **Единый опыт во всех каналах.** Один и тот же SDK даёт одинаковый UX в нативном приложении, в приложении клиента, в Telegram mini apps, MAX и Mattermost — пользователь не «переучивается» при смене канала. При проработке UX нужно уделить отдельное внимание UX - определенные жесты, свайпы, лонгтапы могут по разному вести себя в разных каналах и приводить к негативным сценариям (для примера: жестом можно свернуть miniapp в ТГ, если у нас часть функционала будет завязана на этот жест, то словим негативный опыт)
- **Сохранение истории и контекста.** История переписки и изменения по объекту доступны независимо от канала — это даёт актуальный контекст по заявке.
- **Кастомизация под брендинг клиента.** Внешний вид (цветовые темы) настраивается под инфраструктуру клиента; основа — поддержка цветовых тем SMP и SDK 2.0 (реализация Ильи).
- **Адаптивность к каналу.** Интерфейс корректно работает в разных оболочках и форм-факторах (нативное приложение, mini app, плагин мессенджера).

## Экран 1. Список объектов (скорее всего «Заявки»)

**Описание:** Стартовый экран со списком активных обращений по IT, доступам и оборудованию. Карточки показывают номер, статус, приоритет, категорию, время обновления, ответственного и SLA. Поиск и фильтры работают сразу.

**Функциональность:**
- Кнопка «Создать». По нажатию открывается форма добавления объектов.
- Заголовок раздела «Заявки» берем из названия класса/типа. Используем либо только Заявки для пилота. Либо классы объектов с жизненным циклом 
 - Подзаголовок с описанием то, что находится в разделе. Опционально, отображается если выведен или заполнен. Текст берем из атрибута типа текст класса/типа из Заголовка.
- Строка поиска объектов. Поиск осуществляем по названию заявки
- Фильтрация по статусам 
  - Статусы берем из ЖЦ класса/типа. 
  - Отображаем все статусы в MVP. Далее можем осуществлять проверку и если в конкретном статусе нет доступных объектов, не отображать статус 
- Сводные счётчики-метрики. Это таблички с некими показателями, которые интересны заявителю. Не войдут в MVP т.к. нет понимания, что туда добавить. 
- Список объектов. В рамках MVP список атрибутов будет захрдкожен. Атрибуты должны быть потенциально в каждом классе с ЖЦ. По нажатию на элмент списка мы проваливаемся на карточку объекта. Атрибуты у объекта в списке
  - Название объекта; Если не влезает, то обрезается 
  - Отвественный. Либо пользователь, либо отдел 
  - Степень вовлеченности (я - автор/исполнитель/подписан итд). Не войдет в MVP т.к. для определения степени нужно зайти в каждую заявку и проверить доступные в мобилке атрибуты. 
  - Статус + цвет. Берем из статусной модели объекта
  - Последняя активность. Не войдет в MVP т.к. нужно либо возвращать данные с сервера, либо с мобилки иметь доступ к истории изменений объекта
  - Доп.атрибуты. Здесь будут перечислены атрибуты, которые мы сможем вывести в том или ином виде
   - Класс объекта 
   - Конечный тип объекта (самый низкий в иерархии)
   - Срок (если предусмотрен классом объекта)
   - Что-нибудь еще можно нагенерировать 
- Нижняя навигация (таб-бар). Что там должно находиться не знаю, поэтому перечислю все возможные варианты 
  - Список объектов 
  - Карточка объекта 
  - Экран настроек 
  - Форма добавления 
  - Экран с общим чатом (для общения с ассистентом)
  - Экран профиля 
  Пока наиболее жизнеспособным наполнением таб.бара выглядит следующее наполнение:
  1. Список объектов 
  2. Чат с ассистентом, через который пользователь сможет добавить объект или узнать статус 
  3. Экран настроек. В настройках можно:
    - отключить пуши
    - открыть карточку профиля
    - разлогиниться 
    - посмотреть инфу о стенде

## Экран 2. Чат с ассистентом (главный гипотеза)

**Описание:** Главный рабочий экран — диалог с ботом IT-поддержки. 
**Идеальный сценарии** Пользовать покрывает свои jobs через этот чат. Доступные действия для MVP:
- Заводит заявку
- Находит заявку
- Узнает статусы своих заявок 
- Узнает апдейты своих заявок 
- Узнает описание заявки. Значение поля "Описание" для класса "Заявка"
- Узнает исполнителя по заявке. Значение поля "Исполнитель" для класса "Заявка"
- Узнает остаток времени по заявке. Значение поля "Остаток времени" для класса "Заявка"
- Изменяет статус заявки
- Редактирует описание. Значение поля "Описание" для класса "Заявка"
- Оставляет комментарий. Создание объекта класса "Комментарий" для объекта "Заявка" 


**Функциональность:**
- Шапка чата с заголовком «Чат с ИИ-Ассистентом».
- Меню действий в чате. В MVP предполагаю действие "Создать заявку вручную". В этом случае пользователь переходит на форму добавления объекта. Это действия нужно на случай, если основной сценарий работы с ИИ-ассистентом не сработал. 
- Разделитель даты в ленте сообщений («Сегодня», либо дата сообщений).
- Список сообщений в чате. Типы сообщений и их поведение описаны в соотвествующем разделе 
- Панель добавления сообщений. На панели есть:
  - Инпут для текст комментария 
  - Кнопка отправки сообщения 
  - Кнопка добавления вложений 
  - Кнопка добавления упоминаний (возможно не войдет в MVP)
  - Кнопка меню действий с ИИ-ассистентом. Возможные действия 
    - "Очистить контекст" на случай, если пользователь зашел в тупик. В этом случае система отправляет запрос на очистку кэша чата. В случае успеха, в чате посередине отображается сообщение о том, что кэш чата очищен. В случае неудачи отображается соотвествующая ошибка в всплывающем снэкбаре 
    - Создать заявку. Запускается сценарий "Добавление заявки" 
    - Найти заявку. Запускается сценарий "Поиск заявки"

## Экран 3. Карточка объекта (детальный просмотр)

**Описание:** Детальная статусная карточка заявки: статус, приоритет, описание, SLA, ответственный, следующий шаг, история изменений, комментарии и файлы. Снизу — поле быстрого комментария.

**Функциональность:**
- Шапка с кнопкой «назад» и пометкой времени обновления.
- Заголовок заявки («VPN не подключается с рабочего ноутбука») и пояснение, что заявка создана из чата (сообщения и вложения сохранены в истории).
- Статусная панель: бейдж статуса («В работе»), бейдж приоритета («Высокий приоритет»), индикатор прогресса.
- Блок ключевых полей:
  - Ответственный (Анна Морозова, Сетевая поддержка);
  - SLA с обратным отсчётом («Сегодня, 18:00 · осталось 1 ч 24 мин»);
  - Категория («Сеть и VPN»);
  - Следующий шаг («Ожидается диагностика»).
- Блок «Описание»:
  - Проблема (текст обращения);
  - Устройство (MacBook Pro, офисная сеть и домашний Wi-Fi);
  - Статусы (цепочка: «В работе», «Ожидает проверки» …).
- Вкладки внизу карточки: «История», «Комментарии», «Файлы».
- Поле быстрого комментария снизу с подсказкой ⌘ и кнопкой отправки.
