# Выбор стратегии: Нативное приложение vs PWA  


---

# 🧭 Карткие итоги для Антона 

- **Если важны скорость запуска и бюджет → PWA**
- **Если важны офлайн, стабильность и сложные сценарии → Натив**
- **В вашем случае → оптимальна гибридная стратегия**
  - PWA для офисных пользователей
  - Натив для выездных специалистов

---

# 📊 1. Суть выбора

| Вопрос | Нативное приложение | PWA |
|-------|--------------------|-----|
| Время запуска | Дольше | Быстрее |
| Стоимость | Выше | Ниже |
| Качество UX | Максимальное | Ограниченное |
| Работа офлайн | Надежная | Ограниченная |
| Доступ к функциям устройства | Полный | Частичный |

---

# 💰 2. Текущая ситуация

## Актуальный контекст:
- Команда: **4 человека**
- Уже есть мобильное приложение
- Платформа **не адаптирована под мобильные устройства**

## Сценарии:

### Вариант A — Продолжаем натив
**Плюсы:**
- Используется уже вложенная разработка
- Нет крупных доработок backend 
- Полный доступ к возможностям устройства

**Минусы:**
- Дороже поддержка (iOS + Android)
- Дольше добавление новых функций
- Требуется отдельная мобильная экспертиза

---

### Вариант B — Делаем PWA с нуля
**Плюсы:**
- Быстрее разработка (1 команда)
- Ниже стоимость
- Единая кодовая база
- Легче масштабировать команду

**Минусы:**
- Нужно **переделать архитектуру UI под мобильный web**
- Ограничения платформы (см. ниже)
- Риски UX и производительности

---

# ⚠️ 3. Критические ограничения PWA (важно для бизнеса)

## ❌ Нельзя реализовать:
- Фоновую геолокацию (трекинг сотрудников)
- Надежную офлайн-работу. Есть риск утраты данных при очистке кэша браузера
- Запись видео внутри приложения
- Фоновые процессы (синхронизация, загрузки)

## ⚠️ Ограничено:
- Push-уведомления (на iOS работает, если добавить ссылку на раб.стол)
- Работа с большими объемами данных
- Производительность сложных интерфейсов

👉 Если эти функции критичны — **PWA не подходит как основное решение**

---

# ⚡ 4. Пользовательский опыт (UX)

| Критерий | Натив | PWA |
|----------|------|-----|
| Скорость работы | Высокая | Средняя |
| Плавность интерфейса | Отличная | Ограниченная |
| Работа с большими списками | Хорошо | Может тормозить |
| Ощущение "настоящего приложения" | Да | Частично |

👉 Для сложных рабочих сценариев (полевые сотрудники) UX в PWA может стать проблемой

---

# 🌐 5. Влияние текущего состояния платформы

Ключевой фактор:

> ❗ Платформа НЕ адаптирована под мобильные экраны

Это означает:
- PWA = фактически **крупные доработки фронтенда**
- Нужно:
  - новый UI/UX
  - архитектура с поддержкой мобильной версии
  - актуализация пользовательских сценариев

👉 Это не "дешёвый переход", а **новый продукт**

---

# 🧱 6. Риски каждого варианта

## Натив:
- Рост стоимости команды
- Дольше time-to-market
- Зависимость от мобильных разработчиков

## PWA:
- Потеря критического функционала
- Риск плохого UX
- Ограничения iOS
- Неочевидные баги браузеров

---

# 🔄 7. Рекомендуемая стратегия (оптимальная)

## ✅ Гибридный подход

### 1. PWA — для офисных пользователей
- Просмотр данных
- Редактирование
- Работа в браузере
- Быстрый доступ без установки

### 2. Натив — для выездных сотрудников
- Офлайн работа
- Фото / видео фиксация
- Геолокация
- Надежные уведомления

---

# 📈 8. Когда выбирать каждый вариант

## Выбираем PWA, если:
- Нужно быстро запуститься
- Ограничен бюджет
- Нет сложных мобильных сценариев
- Пользователи работают в офисе

## Выбираем Натив, если:
- Есть выездные сотрудники
- Нужен офлайн режим
- Требуется работа с камерой / гео
- Важен стабильный UX

---

# 🧠 9. Итоговая рекомендация

С учётом:
- текущей команды (4 человека)
- отсутствия мобильной адаптации платформы
- вероятных полевых сценариев

👉 **Не рекомендуется полный переход на PWA**

### Лучшее решение:
1. **Сохранить и развивать нативное приложение**
2. Параллельно:
   - развивать web (PWA) для простых сценариев
   - адаптировать платформу под мобильные экраны

---

# 🧾 Финальный вывод

- PWA = быстрый и дешевый вход, но с ограничениями  
- Натив = дороже, но обеспечивает полный функционал  
- Полный переход на PWA = **риск потери ключевых возможностей продукта**
